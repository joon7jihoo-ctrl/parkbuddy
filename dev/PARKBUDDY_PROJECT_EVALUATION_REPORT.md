# ParkBuddy 프로젝트 평가분석보고서

작성일: 2026-06-18  
대상: ParkBuddy Next.js + Supabase 운영 개발본  
범위: 보안성, 코드 품질/아키텍처, 성능/모바일 UX

## 1. 종합 평가

ParkBuddy는 현재 **운영 가능한 클럽 관리 웹앱의 핵심 흐름**을 대부분 갖춘 상태다. 회원 관리, 일정 투표, 일정 기반 라운딩 생성, 라운딩 참가자/조 편성/스코어 입력/결과 확인, 게시판 보안, 마이페이지 기록 확인까지 주요 기능 흐름이 연결되어 있다.

현재 프로젝트의 가장 큰 장점은 다음과 같다.

- Supabase RLS 기반 권한 모델이 전반적으로 적용되어 있다.
- 운영자 전용 기능은 Server Action과 RPC에서 추가 검증하고 있다.
- 스코어/통계 기준을 `round_scores` 중심으로 통일했다.
- 모바일 운영자를 위한 Dense UX 원칙이 주요 화면에 반영되었다.
- 카카오 로그인 장애를 고려해 초대 코드 기반 백업 로그인 흐름을 마련했다.

현재 남은 가장 큰 리스크는 다음과 같다.

- 전체 검증은 로컬 PC에서 `npm run verify`로 반드시 재확인해야 한다.
- Supabase SQL/RLS는 실제 운영 DB에서 최신 마이그레이션이 모두 적용되어 있어야 한다.
- 패스키 로그인은 아직 준비/검토 단계이며 운영 인증 수단으로 완성된 상태는 아니다.
- 조 편성 알고리즘은 실제 회원 데이터 규모와 팀/소속 데이터 유무를 기준으로 추가 검증이 필요하다.

종합 판단:

```text
운영 전 QA 단계 진입 가능
단, Supabase 설정/마이그레이션/모바일 실기기 테스트 완료 후 운영 권장
```

## 2. 보안성 검토

### 2.1 RLS 적용 상태

Supabase 마이그레이션 기준으로 주요 테이블에 Row Level Security가 적용되어 있다.

확인된 주요 대상:

- `clubs`
- `members`
- `events`
- `event_votes`
- `rounds`
- `round_players`
- `round_participants`
- `round_groups`
- `round_group_members`
- `round_scores`
- `hole_scores`
- `posts`
- `post_attachments`
- `storage.objects`

권한 모델은 대체로 다음 원칙을 따른다.

```text
회원: 같은 클럽 데이터 조회 중심
운영자: 같은 클럽 데이터 생성/수정/관리
작성자: 본인 게시글 제한 수정/삭제
비공개 게시글: 작성자 또는 운영자만 조회
```

### 2.2 권한 탈취 방지

긍정적인 부분:

- `current_member_id()`, `current_club_id()`, `is_club_admin()` 같은 helper function으로 현재 인증 사용자의 클럽/역할을 판단한다.
- `members` 수정 시 `role`, `club_id`, `user_id`, `status` 같은 보호 필드를 일반 회원이 바꾸지 못하도록 trigger가 있다.
- 게시글도 `post_type`, `is_pinned`, `author_id`, `club_id` 같은 보호 필드를 일반 회원이 임의 변경하지 못하도록 trigger가 있다.
- 운영자 RPC 대부분이 `auth.uid()` 확인, active admin 확인, club boundary 확인을 수행한다.
- 게시판 비공개 글과 첨부파일/스토리지 정책이 함께 강화되어, 비공개 글 파일 노출 위험을 줄였다.
- `service_role` 키는 클라이언트 코드에서 발견되지 않았다.

주의할 부분:

- `security definer` RPC는 강력한 권한으로 실행되므로, 모든 함수에 `set search_path = public`이 유지되어야 한다.
- 관리자 RPC가 여러 번 추가되었으므로, 신규 RPC 추가 시 반드시 `AUTH_REQUIRED`, `ADMIN_REQUIRED`, club_id 검증을 반복해야 한다.
- Anonymous sign-ins를 초대 코드 로그인에 사용하므로, 초대 코드 검증 RPC가 운영자/회원 경계와 사용 횟수/만료 조건을 엄격히 지켜야 한다.

### 2.3 게시판/파일 보안

현재 구조는 게시글 visibility와 파일 visibility를 함께 묶는 방향으로 설계되어 있다.

좋은 점:

- 비공개 게시글은 작성자 또는 운영자만 조회 가능하다.
- 첨부파일 조회 정책도 원본 게시글의 공개/비공개 상태를 확인한다.
- Storage path가 `club_id/post_id/file` 구조를 따르도록 정책이 설계되어 있다.
- 이미지 파일 업로드는 파일 타입 검증 유틸을 거친다.

추가 권장:

- 운영 DB에서 `PARKBUDDY_BOARD_SECURITY_SQL_CHECK.sql` 재실행.
- 실제 비공개 게시글 작성자/타회원/운영자 3계정으로 화면 확인.
- 오래된 공개 정책이 남아 있지 않은지 Supabase SQL Editor에서 점검.

### 2.4 보안 평가

| 항목 | 평가 | 비고 |
|---|---:|---|
| RLS 기본 적용 | 좋음 | 주요 테이블에 적용됨 |
| 클럽 경계 보호 | 좋음 | current_club_id 기반 |
| 운영자 권한 보호 | 좋음 | requireAdmin + RPC 검증 |
| 게시판 비공개 보안 | 좋음 | 글/첨부/스토리지 연동 |
| 인증 가용성 | 보통 이상 | 카카오 + 초대 코드, 패스키는 미완성 |
| 운영 DB 검증 필요 | 높음 | 실제 Supabase 상태 확인 필수 |

## 3. 코드 품질 및 아키텍처 검토

### 3.1 Next.js App Router 구조

현재 프로젝트는 Next.js App Router 구조를 잘 활용하고 있다.

좋은 점:

- 화면 단위는 `src/app` 아래 route segment로 구분되어 있다.
- 데이터 조회가 많은 화면은 Server Component 중심으로 구성되어 있다.
- 인터랙션이 필요한 부분만 Client Component로 분리되어 있다.
- Server Action은 `src/server/actions` 또는 route 인접 `actions.ts`에 분리되어 있다.
- Supabase client는 server/client/middleware 파일로 분리되어 있다.

확인된 Client Component 예:

- 카카오 로그인 버튼
- 일정 투표 버튼
- 참가자 선택 enhancer
- 조 편성 form
- 스코어 입력 form
- 복사/공유/출력 버튼
- 모바일 하단 내비게이션
- 스코어 그래프

이 분리는 적절하다. 페이지 전체를 불필요하게 Client Component로 만들지 않고, 사용자 입력과 상태 변화가 필요한 영역만 클라이언트화한 점이 좋다.

### 3.2 재사용성

재사용성이 좋은 영역:

- `src/lib/score-records.ts`: 스코어 기록 기준 통일.
- `src/lib/round-game-labels.ts`: 경기 형태/점수 계산 표시 한글화.
- `src/lib/round-pairing-algorithm.ts`: 조 편성 알고리즘 분리.
- `src/lib/round-linked-event-context.ts`: 라운딩-일정 연결 정보 조회 분리.
- 공통 UI 버튼/복사/프린트/공유 컴포넌트.

추가 개선 여지:

- admin round 계열 화면에 반복되는 header/card/action button 구조는 공통 컴포넌트화 가능하다.
- `dev` 폴더에 과거 개발 문서가 많아져서, 나중에는 `dev/archive`와 `dev/current`로 분리하는 것이 좋다.
- 화면 문구 한글화 매핑이 늘어나면 `src/lib/ui-labels.ts` 같은 중앙 관리 파일로 확장할 수 있다.

### 3.3 가독성과 유지보수성

좋은 점:

- Server Action에서 입력값을 zod 또는 명시적인 파싱으로 검증한다.
- 중요한 mutation은 Supabase RPC로 위임하고 있다.
- 문서가 상세히 축적되어 세션이 끊겨도 개발 흐름 복구가 가능하다.

주의할 점:

- 기능이 빠르게 늘어나면서 일부 화면 파일이 길어졌다.
- 조 편성/스코어/결과 화면은 더 커지기 전에 하위 컴포넌트 분리를 권장한다.
- `dev` 문서가 많아져 현재 작업 우선순위 문서와 과거 문서가 섞여 보일 수 있다.

코드 품질 종합 등급:

```text
B+ / 리팩터링 1회 후 A- 가능
```

## 4. 성능 및 UX 최적화 검토

### 4.1 렌더링 구조

긍정적인 부분:

- 주요 목록/상세 화면은 Server Component 기반 데이터 조회를 사용한다.
- 사용자 입력/선택/필터링이 필요한 영역만 Client Component로 분리되어 있다.
- 스코어 입력, 참가자 선택, 조 편성처럼 상태가 많은 화면은 독립 컴포넌트로 분리되어 있다.

주의할 부분:

- 조 편성/스코어 입력 화면은 참가자 수가 많아지면 Client Component 상태 업데이트가 많아질 수 있다.
- 검색/필터 UI는 회원 수가 커질수록 memoization 또는 서버 검색으로 전환할 수 있다.
- 스코어 그래프는 데이터 수가 커질 경우 페이지 단위 조회 또는 최근 N개 제한이 필요하다.

### 4.2 모바일 Dense UX

반영된 부분:

- 홈/게시판/회원/일정/라운딩/참가자/조 편성/스코어/결과 화면에서 불필요한 설명 문구를 줄였다.
- 라운딩 목록은 compact 카드와 sticky 상태 필터를 적용했다.
- 참가자/조 편성 화면은 모바일에서 더 많은 정보를 한 화면에 볼 수 있도록 grid 밀도를 높였다.
- 스코어 입력 화면은 완료 카드 자동 compact, sticky summary, 누락 필터 등으로 운영 입력 효율을 높였다.
- 모션은 과한 전환 대신 아주 약한 모바일 중심 움직임으로 조정했다.

주의할 부분:

- 모바일 3열/6열/9열 조 편성 카드가 실제 고령 사용자에게 너무 작게 느껴질 수 있다.
- Dense UX는 정보량을 줄이지만, 터치 영역은 최소 44px 기준을 계속 유지해야 한다.
- 설명을 많이 제거했으므로 신규 사용자에게 헷갈리는 지점은 실제 QA에서 다시 보완해야 한다.

### 4.3 사용자 경험 일관성

좋은 방향:

- “최근 공지”처럼 같은 기능을 다른 이름으로 부르는 표현을 “게시판” 기준으로 통일했다.
- 화면에서 `ParkBuddy` 외 영어 노출을 줄이는 방향을 잡았다.
- “자세히 보기”처럼 카드 클릭과 중복되는 문구를 제거했다.
- 라운드 목록에는 “경기 방식”으로 경기 형태와 점수 계산을 함께 표시한다.

추가 확인 필요:

- 아직 일부 화면에 영어/불필요한 설명이 남아 있을 수 있다.
- 실제 모바일 화면 녹화로 “헷갈리는 문구”와 “너무 작은 터치 요소”를 다시 잡아야 한다.

## 5. 핵심 리스크와 권장 조치

### 최우선 QA

1. Supabase 마이그레이션 최신 적용 여부 확인.
2. `npm run verify` 전체 통과 확인.
3. 운영자 계정으로 `/admin/rounds` → 참가자 → 조 편성 → 스코어 → 결과 흐름 확인.
4. 일반 회원 계정으로 `/schedule`, `/scores`, `/mypage`, `/board` 권한 확인.
5. 비공개 게시글과 첨부파일 권한 확인.
6. 모바일 데이터망에서 카카오 로그인 실패 시 초대 코드 경로 확인.

### 다음 개발 우선순위

1. 운영 QA에서 발견된 오류 수정.
2. 조 편성 알고리즘 실제 데이터 검증.
3. 회원 팀/소속 데이터 구조 설계.
4. 패스키 도입은 운영 도메인 확정 후 재검토.
5. 과거 dev 문서 archive 정리.

## 6. 최종 결론

ParkBuddy는 현재 “계속 기능을 추가하는 단계”보다 **운영 QA와 안정화 단계**에 들어가는 것이 맞다.

현 시점에서 가장 중요한 원칙은 다음이다.

```text
새 기능보다 실제 사용자 흐름을 끝까지 검증한다.
스코어와 통계는 round_scores 기준을 유지한다.
권한은 Supabase RLS와 RPC에서 이중으로 방어한다.
화면은 짧고 조용하게 유지하되, 헷갈리는 지점만 설명한다.
```

이 기준으로 운영 QA를 통과하면 ParkBuddy는 실제 클럽 운영 테스트에 들어갈 수 있는 수준이다.
